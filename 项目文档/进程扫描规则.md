# 进程扫描规则与评分原理

> ProcRisk 基于规则引擎模式实现进程风险检测。系统包含两种扫描模式（全量采集与单进程检测），5 条内置检测规则，1 套 Beaconing 周期性通信检测算法（含多轮采样与代理穿透），以及基于维度加法的评分机制。

---

## 一、扫描模式

### 1.1 全量进程采集（首页「扫描进程」按钮）

**触发方式**：`POST /api/scan-processes`

**流程**：

1. 调用 `psutil.process_iter()` 两轮遍历本机所有进程（上限 5000）
   - 第一轮：构建 `pid → name` 映射缓存，用于解析父进程名
   - 第二轮：逐进程采集 PID、名称、路径、命令行、用户名、父进程、启动时间、CPU/内存占用、网络连接等
2. 对每个进程打上静态风险标签：
   - `is_system`：路径是否属于 Windows 系统目录
   - `signed_status`：系统路径进程启发式标记为 "signed"，其余为 "unknown"
   - `path_suspicious`：路径是否匹配可疑模式（Temp、Downloads、Desktop、Public 等 9 种模式）
   - `parent_child_suspicious`：父子进程是否属于已知可疑组合
3. 调用 `sync_process_catalog()` 将进程信息写入 `process_catalog` 表（按 `process_key` 去重，已存在则更新 `last_seen_at` 及风险标签）
4. 创建 `ScanRun` + `ScanTarget` 关联记录

**注意**：全量采集仅存储进程快照，**不执行规则引擎检测和评分**。评分需在进程详情页点击「扫描此进程」触发单进程检测。

### 1.2 单进程检测（进程详情页「扫描此进程」按钮）

**触发方式**：`POST /scans/api/start`（异步 API）

**流程**（`detect_process()` 流水线）：

```
┌──────────────────────────────────────────────────────────┐
│  1. 采集进程信息 (get_process_by_pid)                      │
│  2. 采集网络连接 (get_connections_for_pid) — 首轮快照       │
│  3. BeaconSampler 多轮采样 (默认 12轮×5秒 = 60秒)          │
│     每轮: get_connections_for_pid() → 记录 timestamp        │
│     对比上一轮: 提取新增远端连接 → check-in 事件             │
│     跟踪远端出现轮次: 检测持久连接                           │
│  4. 加载白名单 + DB 规则配置                                │
│  5. 构建 RuleContext                                       │
│  6. RuleEngine.evaluate() → 命中 PROC 类规则               │
│  7. detect_beaconing_from_sampling() → Beaconing 检测结果   │
│  8. 注入 beacon_stats → 再次 evaluate() → 命中 NET-101     │
│  9. compute_score() → 汇总评分                              │
│ 10. 聚合远端 IP 汇总（含代理穿透）                           │
│ 11. 写入 DB (ScanRun → ScanTarget → 6 张子表)              │
└──────────────────────────────────────────────────────────┘
```

**异步扫描机制**：前端通过 `POST /scans/api/start` 启动后台线程，返回 `task_id`；前端每秒轮询 `GET /scans/api/status/<task_id>` 获取进度（采样轮次/总轮次），完成后自动跳转扫描详情页。采样期间前端展示进度条，避免页面卡顿。

**白名单前置过滤**：若进程命中白名单（名称精确匹配或路径前缀匹配），则直接跳过所有规则检测，评分结果为 0 分 / LOW。

## 二、检测规则详情

### 2.1 PROC-001：异常父子进程关系

| 属性 | 值 |
|------|-----|
| 维度 | process |
| 默认权重 | 20 分 |
| 检测逻辑 | 父进程名在 `SUSPICIOUS_PARENT_CHILD` 映射表中，且子进程名在对应的可疑子进程列表中 |

**已知可疑父子组合**：

| 父进程 | 可疑子进程 |
|--------|-----------|
| `svchost.exe` | cmd.exe, powershell.exe, wscript.exe, cscript.exe |
| `taskeng.exe` | cmd.exe, powershell.exe |
| `mshta.exe` | cmd.exe, powershell.exe |
| `winword.exe` | cmd.exe, powershell.exe, wscript.exe |
| `excel.exe` | cmd.exe, powershell.exe |
| `outlook.exe` | cmd.exe, powershell.exe |

**原理**：正常情况下 `svchost.exe` 作为服务宿主不应直接启动命令行或脚本解释器。若观察到此类父子关系，通常意味着恶意代码注入或漏洞利用（如 Office 宏执行 PowerShell）。匹配时不区分大小写。

**证据示例**：
```json
{
  "parent": "svchost.exe",
  "child": "cmd.exe",
  "detail": "svchost.exe -> cmd.exe 属于可疑父子组合"
}
```

### 2.2 PROC-002：运行于可疑路径

| 属性 | 值 |
|------|-----|
| 维度 | process |
| 默认权重 | 30 分 |
| 检测逻辑 | 可执行文件路径匹配可疑路径模式，**且**签名状态不为 "signed" |

**可疑路径模式**：

| 模式 | 典型含义 |
|------|---------|
| `\Temp\` | 临时目录，恶意软件常用释放位置 |
| `\AppData\Local\Temp\` | 用户临时目录，无需管理员权限即可写入 |
| `\Downloads\` | 浏览器下载目录，钓鱼/社工投递目标 |
| `\Public\` | 公共目录，多用户可访问 |
| `\Desktop\` | 桌面目录，用户直接下载运行的位置 |
| `\AppData\Roaming\` | 漫游数据目录，攻击者持久化常用路径 |
| `\ProgramData\` | 全局程序数据目录，无需管理员权限可写 |
| `\PerfLogs\` | 性能日志目录，系统默认为空，常被滥用 |
| `\Recycle\` | 回收站目录，攻击者藏匿文件 |

**原理**：合法软件通常安装在 `Program Files` 或 `Windows\System32` 等标准路径下，且具备有效数字签名。在临时目录、桌面、下载目录等用户可写区域运行的未签名可执行文件，高度可疑——这是恶意软件规避检测的常见手法（投递→释放→执行）。

**证据示例**：
```json
{
  "path": "C:\\Users\\test\\Desktop\\artifact_x64.exe",
  "matched_patterns": ["\\Desktop\\"],
  "signed_status": "unknown",
  "detail": "路径包含 \\Desktop\\ 且签名状态为 unknown"
}
```

### 2.3 PROC-003：未签名的非系统进程

| 属性 | 值 |
|------|-----|
| 维度 | process |
| 默认权重 | 15 分 |
| 检测逻辑 | 签名状态不为 "signed"，且不属于系统路径 |

**原理**：Windows 系统中，合法的商业软件几乎都包含 Authenticode 数字签名。非系统路径下且非 "signed" 状态的进程，缺乏身份可信度。该规则权重低于 PROC-002（15 vs 30），因为单独的未签名状态仅表示"可信度不足"，需结合其他指标（如可疑路径）才能构成强风险信号。

**注意**：当前系统 `signed_status` 为启发式赋值（系统路径→"signed"，其余→"unknown"），未进行真实数字签名验证。因此 PROC-003 实际检测的是"非系统路径的非系统进程"。

**证据示例**：
```json
{
  "name": "suspicious_tool.exe",
  "path": "C:\\Tools\\suspicious_tool.exe",
  "signed_status": "unknown",
  "detail": "签名状态为 unknown，非系统路径"
}
```

### 2.4 PROC-004：可疑进程名

| 属性 | 值 |
|------|-----|
| 维度 | process |
| 默认权重 | 20 分 |
| 检测逻辑 | 进程名匹配已知恶意工具命名模式或疑似随机命名 |

**已知恶意命名模式**：

| 模式 | 典型含义 |
|------|---------|
| `artifact*` | Cobalt Strike 生成的 payload |
| `beacon*` | C2 Beacon |
| `payload*` | 通用 payload |
| `shellcode*` | Shellcode 加载器 |
| `inject*` | 注入器 |
| `mimikatz*` | 凭证转储工具 |
| `procdump*` | 进程内存转储 |
| `lazagne*` | 密码恢复工具 |
| `nmap*` / `ncat*` | 网络扫描/后门 |
| `psexec*` | 远程执行工具 |

**随机命名检测**：8 位以上纯字母数字组成的 `.exe` 文件名（如 `xjkdqwer.exe`），常见于自动生成的恶意样本。

**原理**：合法商业软件通常有有意义的命名（如 `chrome.exe`、`notepad.exe`），而攻击工具和自动生成的 payload 往往使用特征性名称或随机字符串。

**证据示例**：
```json
{
  "name": "artifact_x64.exe",
  "path": "C:\\Users\\test\\Desktop\\artifact_x64.exe",
  "reasons": ["匹配已知恶意模式 ^artifact"],
  "detail": "进程名 artifact_x64.exe: 匹配已知恶意模式 ^artifact"
}
```

### 2.5 NET-101：疑似 Beaconing 周期性通信

| 属性 | 值 |
|------|-----|
| 维度 | network |
| 默认权重 | 35 分 |
| 检测逻辑 | `beacon_stats.suspected == True`（由 Beaconing 检测模块注入上下文） |

**原理**：Beaconing 是 C2（命令与控制）通信的典型行为特征——被控主机以固定或近似固定的时间间隔向控制服务器发送心跳包。NET-101 规则本身不执行检测算法，而是读取 `detect_beaconing_from_sampling()` 的结果，当判定为疑似 Beaconing 时触发。该规则权重最高（35 分），因为 Beaconing 行为是主机被植入后门的最强指标之一。

**证据示例**：
```json
{
  "sample_window": 12,
  "interval_mean_ms": 5000.0,
  "interval_cv": 0.08,
  "jitter_like": false,
  "detail": "CV=0.080, 均值=5000.0ms"
}
```

## 三、Beaconing 检测算法

### 3.1 多轮采样机制

**核心问题**：`psutil` 提供的是瞬时快照，单次采集所有连接的 `seen_at` 相同，无法计算时间间隔。这使得纯 CV 算法在单次快照场景下完全失效。

**解决方案**：引入多轮采样机制（`BeaconSampler`），在检测前以固定间隔多次采集进程的网络连接快照，积累真正的时间序列数据。

**采样流程**：

```
┌────────────────────────────────────────────────────────┐
│  用户点击「扫描此进程」                                   │
│       ↓                                                 │
│  1. 采集进程信息 + 首轮连接快照                           │
│  2. BeaconSampler 启动（默认 12 轮 × 5 秒 = 60 秒）      │
│     每轮: get_connections_for_pid() → 记录 timestamp      │
│     对比上一轮: 提取新增远端连接 → check-in 事件           │
│     跟踪远端出现轮次: 检测持久连接                         │
│  3. 返回 SamplingResult (checkin_times + persistent)     │
│  4. detect_beaconing_from_sampling() 分析               │
└────────────────────────────────────────────────────────┘
```

**关键概念**：

| 概念 | 定义 | 检测意义 |
|------|------|----------|
| **Check-in** | 某轮采样新出现的远端连接（上一轮不存在） | Beacon 每次回连 C2 就是一次 check-in |
| **持久连接** | 在 ≥80% 采样轮次中持续存在的远端连接 | 长连接模式的 Beacon（如 HTTP 长轮询） |

### 3.2 基于 Check-in 间隔的 CV 检测

**核心思想**：Beaconing 通信的 check-in 间隔高度规律，变异系数（CV = 标准差 / 均值）接近 0。正常网络流量间隔波动较大，CV 通常 > 0.3。

**算法步骤**：

1. 从多轮采样结果提取 `checkin_times` 列表
2. 计算相邻 check-in 的时间间隔（毫秒）
3. 若间隔数 < `BEACON_MIN_SAMPLES`（10），降级处理
4. 计算统计量：
   - 均值 μ = Σinterval / N
   - 方差 σ² = Σ(interval - μ)² / N
   - 标准差 σ = √σ²
   - 变异系数 CV = σ / μ
5. 判定：
   - **CV < 0.3** → `suspected = True`（疑似 Beaconing）
   - **0.05 < CV < 0.15** → `jitter_like = True`（含 Jitter 特征，攻击者可能添加随机延迟规避检测）

**弱信号处理**：当 check-in 次数不足 CV 阈值（10 次）但 ≥ 2 次时，若 check-in 频率 ≥ 50% 采样轮次，仍标记为 `suspected = True`。

### 3.3 持久连接检测

某些 Beacon 使用 HTTP 长轮询或 WebSocket 保持持久连接，而非周期性新建连接。这种模式下没有明显的 check-in 事件，但存在一个始终活跃的远端连接。

**检测逻辑**：统计每个 `(remote_ip, remote_port)` 对在采样轮次中的出现次数，若出现频率 ≥ 80%，则标记为「持久连接」并触发 `suspected = True`。

### 3.4 代理穿透

当被控主机通过本地代理（如 Clash、Shadowsocks）通信时，`psutil` 报告的远端 IP 为回环地址（如 `127.0.0.1:7897`），真实 C2 地址被代理隐藏。

**代理穿透流程**（`find_proxy_connections()`）：

```
┌───────────────────────────────────────────────────────┐
│  发现远端 IP 为 127.0.0.1:7897                         │
│       ↓                                                │
│  1. 遍历 psutil.net_connections()                      │
│     找到 LISTEN 状态、端口 7897、绑定回环的连接           │
│  2. 获取该连接对应的进程 PID 和名称（如 verge-mihomo.exe）│
│  3. 采集该代理进程的远端连接（排除回环地址）              │
│  4. 去重 (ip, port)，最多返回 10 条                     │
│  5. 在结果中添加 _proxy_via 和 _proxy_local 元数据      │
│       ↓                                                │
│  远端 IP 汇总中用代理真实远端替换回环地址                 │
│  risk_hint 标注"经代理 verge-mihomo.exe(PID xxx)"       │
└───────────────────────────────────────────────────────┘
```

**回环地址集合**：`{127.0.0.1, 0.0.0.0, ::1, ::}`

### 3.5 启发式补充（单次快照回退模式）

当多轮采样不可用时（如配置关闭或无连接），系统回退到单次快照启发式：

- **同 IP 多连接**：同一远端 IP 有 ≥ 2 条并发连接 → 可疑
- **非标准端口**：远端端口不在常见端口集合中 → 可疑

常见端口白名单：{20, 21, 22, 23, 25, 53, 80, 110, 143, 443, 465, 587, 993, 995, 3306, 3389, 5432, 8080, 8443}

### 3.6 参数配置

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `BEACON_MIN_SAMPLES` | 10 | 最少间隔样本数，不足则不计算 CV |
| `BEACON_CV_THRESHOLD` | 0.3 | CV 低于此值判定为疑似 Beaconing |
| `BEACON_JITTER_CV_MIN` | 0.05 | Jitter 检测的 CV 下限 |
| `BEACON_SAMPLING_ENABLED` | True | 是否启用多轮采样 |
| `BEACON_SAMPLING_INTERVAL` | 5 | 采样间隔（秒） |
| `BEACON_SAMPLING_ROUNDS` | 12 | 采样轮数（总时长 = interval × rounds） |

以上参数均可在 `config.py` 中配置。

## 四、评分机制

### 4.1 评分算法

评分采用**维度加法模型**：将所有命中规则的 `score_delta` 按维度分组求和，总分上限 100。

```
total_score = min(score_process + score_network + score_memory + score_other, 100)
```

**维度映射**：

| 规则 | 维度 | 对应子分字段 |
|------|------|-------------|
| PROC-001 | process | `score_process` |
| PROC-002 | process | `score_process` |
| PROC-003 | process | `score_process` |
| PROC-004 | process | `score_process` |
| NET-101 | network | `score_network` |

### 4.2 风险等级划分

| 等级 | 分数范围 | 含义 |
|------|---------|------|
| LOW | 0 - 39 | 无明显风险特征 |
| MID | 40 - 69 | 存在部分可疑特征，需关注 |
| HIGH | 70 - 100 | 多个高风险特征叠加，需优先处置 |

### 4.3 评分示例

**场景 A：系统进程 svchost.exe**

| 检查项 | 结果 |
|--------|------|
| 白名单 | 命中（svchost.exe 在默认白名单中） |
| 规则检测 | 跳过（白名单前置过滤） |
| **总分** | **0 / LOW** |

**场景 B：Cobalt Strike Beacon（artifact_x64.exe）**

| 规则 | 是否命中 | 得分 |
|------|---------|------|
| PROC-001（异常父子） | 取决于父进程 | 0 或 +20 |
| PROC-002（可疑路径） | 路径含 `\Desktop\`，未签名，命中 | +30 |
| PROC-003（未签名非系统） | 签名状态 unknown，非系统路径，命中 | +15 |
| PROC-004（可疑进程名） | 匹配 `^artifact` 模式，命中 | +20 |
| NET-101（Beaconing） | 多轮采样检测到持久连接或周期性 check-in，命中 | +35 |
| **合计** | | **100（封顶）/ HIGH** |

**场景 C：用户目录下的普通程序**

| 规则 | 是否命中 | 得分 |
|------|---------|------|
| PROC-001 | 父进程为 explorer.exe，不在可疑映射中 | 0 |
| PROC-002 | 路径不含可疑模式 | 0 |
| PROC-003 | 签名状态 unknown，非系统路径，命中 | +15 |
| PROC-004 | 进程名正常 | 0 |
| NET-101 | 无网络连接 | 0 |
| **合计** | | **15 / LOW** |

### 4.4 摘要生成

`summary_reason` 字段由命中规则标题按分数降序拼接而成，截断至 255 字符：

```
"疑似 Beaconing 周期性通信 + 运行于可疑路径 + 异常父子进程关系 + 未签名的非系统进程"
```

## 五、规则引擎架构

### 5.1 类注册机制

系统采用 `@register_rule` 装饰器 + 全局注册表模式，而非 YAML 配置文件：

```python
@register_rule
class MyCustomRule(BaseRule):
    rule_id = "PROC-005"
    title = "自定义规则"
    dimension = "process"
    default_weight = 25

    def check(self, context: RuleContext) -> Optional[RuleHitResult]:
        # 实现检测逻辑
        ...
```

`RuleEngine` 初始化时自动实例化所有 `@register_rule` 注册的规则类，无需手动配置。

### 5.2 数据库配置覆盖

`RuleEngine.load_db_config(db_session)` 从 `rules` 表读取配置：

- 若 DB 中 `enabled = False`，该规则从引擎中移除
- 若 DB 中 `default_weight` 与代码默认值不同，使用 DB 中的值覆盖
- 若 DB 中有规则但代码未实现，创建占位规则（`check()` 永远返回 None）

### 5.3 白名单过滤

规则引擎在执行前首先检查白名单。若进程匹配任意一条启用的白名单条目，则跳过所有规则检测：

| 白名单类型 | 匹配方式 | 示例 |
|-----------|---------|------|
| `process_name` | 进程名精确匹配（不区分大小写） | `svchost.exe` |
| `process_path` | 路径前缀匹配（不区分大小写） | `C:\Windows\System32\` |
| `remote_ip` | 远端 IP 精确匹配 | `8.8.8.8` |
| `rule_id` | 在规则引擎中处理（预留） | `NET-101` |

## 六、扩展指南

### 添加新检测规则

1. 在 `utils/rule_engine/rules.py` 中创建继承 `BaseRule` 的类
2. 添加 `@register_rule` 装饰器
3. 设置 `rule_id`、`title`、`dimension`、`default_weight`
4. 实现 `check(self, context: RuleContext) -> Optional[RuleHitResult]`
5. 规则引擎自动发现并加载，无需修改其他代码

### 添加新检测维度

当前支持 4 个维度：`process`、`network`、`memory`、`other`。添加新维度需同步修改：

1. `utils/common/constants.py` 中的 `DIMENSIONS` 元组
2. `webapp/models.py` 中 `Rule.dimension` 的 Enum 定义
3. `webapp/models.py` 中 `RuleHit.dimension` 的 Enum 定义
4. `utils/risk_detector/scorer.py` 中 `ScoreResult` 的子分字段和 `compute_score()` 的维度映射
5. 数据库 ALTER TABLE 更新 Enum 值
